The application uses **NeonDB** (serverless PostgreSQL) with **Drizzle ORM** for type-safe database operations. All schema definitions are in `lib/db/schema.ts`.

## Database overview

The schema consists of three main groups:

1. **Authentication tables** - NextAuth.js user management (users, accounts, sessions, verificationTokens)
2. **Application tables** - Core features (favorites, dailyLogs, emailPreferences)
3. **Reaction tables** - Anonymous and authenticated reactions (menuReactions, userReactions)

## Schema definition

Schemas are defined using Drizzle ORM's PostgreSQL table builder:

```typescript
import {
  pgTable,
  text,
  timestamp,
  integer,
  primaryKey,
  jsonb,
  date,
  serial,
  uniqueIndex,
  boolean,
} from "drizzle-orm/pg-core";
```

## Authentication tables

### users

Stores user account information managed by NextAuth.js:

```typescript
export const users = pgTable("users", {
  id: text("id")
    .primaryKey()
    .$defaultFn(() => crypto.randomUUID()),
  name: text("name"),
  email: text("email").unique(),
  emailVerified: timestamp("emailVerified", { mode: "date" }),
  image: text("image"),
  calorieGoal: integer("calorie_goal"),
});
```

### accounts

Stores OAuth account connections (Google):

```typescript
export const accounts = pgTable(
  "accounts",
  {
    userId: text("userId")
      .notNull()
      .references(() => users.id, { onDelete: "cascade" }),
    type: text("type").$type<AdapterAccountType>().notNull(),
    provider: text("provider").notNull(),
    providerAccountId: text("providerAccountId").notNull(),
    refresh_token: text("refresh_token"),
    access_token: text("access_token"),

    token_type: text("token_type"),
    scope: text("scope"),

    session_state: text("session_state"),
  },
  (account) => [
    primaryKey({
      columns: [account.provider, account.providerAccountId],
```

## Application tables

### favorites

Stores user's favorite meals for notifications:

```typescript
export const favorites = pgTable(
  "favorites",
  {
    id: serial("id").primaryKey(),
    userId: text("user_id")
      .notNull()
      .references(() => users.id, { onDelete: "cascade" }),
    mealId: text("meal_id"),
    mealName: text("meal_name").notNull(),
    createdAt: timestamp("created_at", { mode: "date" }).defaultNow().notNull(),
  },
  (table) => [
    uniqueIndex("favorites_user_meal_idx").on(table.userId, table.mealName),
  ]
);
```

<AccordionGroup>

<Accordion title="favorites table schema">

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | serial | PRIMARY KEY | Auto-increment ID |
| `userId` | text | FOREIGN KEY → users.id, CASCADE, NOT NULL | User who favorited |
| `mealId` | text | nullable | Meal ID from cafeteria system |
| `mealName` | text | NOT NULL | Meal name for matching |
| `createdAt` | timestamp | DEFAULT NOW(), NOT NULL | When favorited |

**Unique index:** `(userId, mealName)` - Prevents duplicate favorites

**Example row:**
```json
{
  "id": 42,
  "userId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "mealId": "157",
  "mealName": "Ekşili Köfte",
  "createdAt": "2026-02-15T14:30:00Z"
}
```

</Accordion>

<Accordion title="Why mealId is nullable">

`mealId` is nullable because:

1. **Meal IDs can change** - University system may reassign IDs
2. **Name-based matching is more reliable** - Users recognize meals by name
3. **Future-proofing** - If ID system changes, favorites still work

The notification system matches by `mealName` when sending daily emails.

</Accordion>

</AccordionGroup>

## Reaction tables

### menuReactions

Stores anonymous like/dislike counts per menu date:

```typescript
export const menuReactions = pgTable("menu_reactions", {
  id: serial("id").primaryKey(),
  menuDate: text("menu_date").unique().notNull(),
  likeCount: integer("like_count").default(0).notNull(),
  dislikeCount: integer("dislike_count").default(0).notNull(),
  legacyLikeCount: integer("legacy_like_count").default(0).notNull(),
  legacyDislikeCount: integer("legacy_dislike_count").default(0).notNull(),
  createdAt: timestamp("created_at", { mode: "date" }).defaultNow(),
  updatedAt: timestamp("updated_at", { mode: "date" }).defaultNow(),
});
```

<AccordionGroup>

<Accordion title="menuReactions table schema">

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | serial | PRIMARY KEY | Auto-increment ID |
| `menuDate` | text | UNIQUE, NOT NULL | Menu date (YYYY-MM-DD) |
| `likeCount` | integer | DEFAULT 0, NOT NULL | Current like count |
| `dislikeCount` | integer | DEFAULT 0, NOT NULL | Current dislike count |
| `legacyLikeCount` | integer | DEFAULT 0, NOT NULL | Historical likes (pre-auth) |
| `legacyDislikeCount` | integer | DEFAULT 0, NOT NULL | Historical dislikes (pre-auth) |
| `createdAt` | timestamp | DEFAULT NOW() | First reaction time |
| `updatedAt` | timestamp | DEFAULT NOW() | Last reaction time |

**Example row:**
```json
{
  "id": 42,
  "menuDate": "2026-02-19",
  "likeCount": 127,
  "dislikeCount": 23,
  "legacyLikeCount": 0,
  "legacyDislikeCount": 0,
  "createdAt": "2026-02-19T06:00:00Z",
  "updatedAt": "2026-02-19T15:30:00Z"
}
```

</Accordion>

<Accordion title="Legacy vs current counts">

**Legacy counts** (`legacyLikeCount`, `legacyDislikeCount`):
- Historical data from before user authentication was implemented
- Preserved for continuity but not actively updated

**Current counts** (`likeCount`, `dislikeCount`):
- Active counters for authenticated reactions
- Can be incremented/decremented based on user actions

</Accordion>

</AccordionGroup>

### userReactions

Tracks individual user reactions:

```typescript
export const userReactions = pgTable(
  "user_reactions",
  {
    id: serial("id").primaryKey(),
    userId: text("user_id")
      .notNull()
      .references(() => users.id, { onDelete: "cascade" }),
    menuDate: text("menu_date").notNull(),
    action: text("action").$type<"like" | "dislike">().notNull(),
    createdAt: timestamp("created_at", { mode: "date" }).defaultNow().notNull(),
    updatedAt: timestamp("updated_at", { mode: "date" }).defaultNow().notNull(),
  },
  (table) => [
    uniqueIndex("user_reactions_user_date_idx").on(table.userId, table.menuDate),
  ]
);
```

<AccordionGroup>

<Accordion title="userReactions table schema">

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | serial | PRIMARY KEY | Auto-increment ID |
| `userId` | text | FOREIGN KEY → users.id, CASCADE, NOT NULL | User who reacted |
| `menuDate` | text | NOT NULL | Menu date (YYYY-MM-DD) |
| `action` | text | NOT NULL, "like" \| "dislike" | Reaction type |
| `createdAt` | timestamp | DEFAULT NOW(), NOT NULL | First reaction time |
| `updatedAt` | timestamp | DEFAULT NOW(), NOT NULL | Last change time |

**Unique index:** `(userId, menuDate)` - One reaction per user per menu date

**Example row:**
```json
{
  "id": 789,
  "userId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "menuDate": "2026-02-19",
  "action": "like",
  "createdAt": "2026-02-19T08:15:00Z",
  "updatedAt": "2026-02-19T08:15:00Z"
}
```

</Accordion>

<Accordion title="Reaction workflow">

**User likes a menu:**
1. Check if `userReactions` entry exists for user + date
2. If exists and action is "like": Remove reaction (toggle off)
3. If exists and action is "dislike": Change to "like", update `menuReactions` counts
4. If not exists: Create "like" entry, increment `menuReactions.likeCount`

**User dislikes a menu:**
- Same logic, but with "dislike" action

</Accordion>

</AccordionGroup>

## Common queries

### Get user's favorites

```typescript
import { db } from '@/lib/db';
import { favorites } from '@/lib/db/schema';
import { eq } from 'drizzle-orm';

const userFavorites = await db
  .select()
  .from(favorites)
  .where(eq(favorites.userId, userId));
```

### Get daily log for specific date

```typescript
import { dailyLogs } from '@/lib/db/schema';
import { and, eq } from 'drizzle-orm';

const log = await db
  .select()
  .from(dailyLogs)
  .where(
    and(
      eq(dailyLogs.userId, userId),
      eq(dailyLogs.date, '2026-02-19')
    )
  )
  .limit(1);
```

### Get menu reactions for date

```typescript
import { menuReactions } from '@/lib/db/schema';

const reactions = await db
  .select()
  .from(menuReactions)
  .where(eq(menuReactions.menuDate, '2026-02-19'))
  .limit(1);
```

### Check if user has reacted to menu

```typescript
import { userReactions } from '@/lib/db/schema';

const reaction = await db
  .select()
  .from(userReactions)
  .where(
    and(
      eq(userReactions.userId, userId),
      eq(userReactions.menuDate, '2026-02-19')
    )
  )
  .limit(1);
```

## Database migrations

The project uses Drizzle Kit for migrations:

```bash
# Generate migration files
npx drizzle-kit generate

# Push schema to database (development)
npx drizzle-kit push

# View database in Drizzle Studio
npx drizzle-kit studio
```

Configuration in `drizzle.config.ts`:

```typescript
import { defineConfig } from 'drizzle-kit';

export default defineConfig({
  schema: './lib/db/schema.ts',
  out: './drizzle',
  dialect: 'postgresql',
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
});
```

## Connection pooling

NeonDB connection setup in `lib/db/index.ts`:

```typescript
import { drizzle } from 'drizzle-orm/neon-http';
import { neon } from '@neondatabase/serverless';
import * as schema from './schema';

const sql = neon(process.env.DATABASE_URL!);
export const db = drizzle(sql, { schema });
```

<Note>
NeonDB uses serverless HTTP connections, which don't require connection pooling in the traditional sense. Each request creates a new HTTP connection.
</Note>

## Type-safe queries

Drizzle provides full TypeScript inference:

```typescript
// ✅ Fully typed
const user = await db
  .select()
  .from(users)
  .where(eq(users.id, userId))
  .limit(1);

// user is typed as:
// { id: string; name: string | null; email: string | null; ... }[]

// ✅ Auto-complete for columns
const favs = await db
  .select({
    name: favorites.mealName,
    created: favorites.createdAt,
  })
  .from(favorites);

// favs is typed as:
// { name: string; created: Date }[]
```

## Performance considerations

- **Indexes**: All foreign keys and frequently queried columns have indexes
- **Unique constraints**: Prevent duplicate entries and improve query performance
- **JSONB**: `consumedMeals` uses JSONB for efficient JSON storage and querying
- **Cascading deletes**: User deletion automatically removes all related data
- **Serverless**: NeonDB scales automatically with traffic
